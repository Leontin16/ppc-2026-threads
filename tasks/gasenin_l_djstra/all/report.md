## 1. Введение

Данный отчёт описывает гибридную реализацию алгоритма Дейкстры, совмещающую MPI
для межпроцессного параллелизма и OpenMP для внутрипроцессного.
MPI распределяет вершины графа между рангами, каждый из которых независимо обрабатывает
свой блок; OpenMP распараллеливает поиск локального минимума и релаксацию внутри
каждого ранга. Последовательная версия (`SEQ`) служит эталоном корректности и ускорения.

## 2. Постановка задачи

Дан неориентированный полносвязный граф с $n$ вершинами.

- Вес ребра между вершинами $u$ и $v$ равен $|u - v|$.
- **Входные данные:** количество вершин $n$.
- **Выходные данные:** сумма длин кратчайших путей от вершины 0 до всех остальных.
- **Ограничения:** $n > 0$. Теоретическая сумма: $\frac{n(n-1)}{2}$.

## 3. Схема параллельного алгоритма

### 3.1 Распределение данных (MPI-уровень)

В `PreProcessingImpl` вершины графа делятся между `size_` рангами блочным
распределением с остатком:

- Ранги с номером `< rem` получают блок размером `chunk + 1`.
- Остальные ранги получают блок размером `chunk`.

Каждый ранг хранит только свой фрагмент векторов `dist_` и `visited_`.
Вершина-источник (0) инициализируется нулём тем рангом, в чей диапазон она попадает.

### 3.2 Итерация алгоритма (два уровня параллелизма)

Каждая из $n$ итераций включает четыре шага:

1. **OpenMP-поиск локального минимума** (`FindThreadLocalMinima`): потоки OpenMP
   независимо проходят свои части локального блока и записывают результаты в
   `thread_mins[tid]` / `thread_vertices[tid]`. Используется `#pragma omp for nowait`.
2. **Свёртка потоковых минимумов** (`ReduceThreadMinima`): главный поток объединяет
   локальные результаты потоков в пару `(local_min, local_vertex)`.
3. **MPI-редукция глобального минимума** (`MPI_Allreduce` с пользовательской операцией
   `min_pair_op`): все ранги атомарно согласовывают глобальную вершину с наименьшим
   расстоянием. Пользовательский оператор `MinPairImpl` сравнивает пары
   `(расстояние, индекс)`.
4. **OpenMP-релаксация** (`UpdateLocalDistances`): директива `#pragma omp parallel for`
   обновляет расстояния до непосещённых локальных вершин. Если глобальная вершина
   принадлежит текущему рангу, она помечается посещённой.

По окончании всех итераций каждый ранг вычисляет локальную сумму достижимых расстояний
(с OpenMP-редукцией) и передаёт её в `MPI_Allreduce(MPI_SUM)`.

### 3.3 Ключевой фрагмент — пользовательская операция MPI

Файл: `all/src/ops_all.cpp`, функция `MinPairImpl`.

```cpp
void MinPairImpl(void *in, void *inout, const int *len, MPI_Datatype * /*dtype*/) {
  auto *a = static_cast<InType *>(in);
  auto *b = static_cast<InType *>(inout);
  for (int i = 0; i < *len; i += 2) {
    if (a[i] < b[i]) {
      b[i]     = a[i];
      b[i + 1] = a[i + 1];
    }
  }
}
```

Операция обрабатывает буфер как массив пар `(расстояние, индекс_вершины)`.
При вызове `MPI_Allreduce` с `count = 1` и данной операцией все ранги получают
одну и ту же глобальную пару-победитель.

### 3.4 Ключевой фрагмент — OpenMP-поиск локального минимума

Файл: `all/src/ops_all.cpp`, функция `FindThreadLocalMinima`.

```cpp
#pragma omp parallel default(none) \
    shared(local_n, start_v, dist, visited, thread_mins, thread_vertices, inf)
{
  const int tid = omp_get_thread_num();
  InType t_min = inf;
  InType t_v   = -1;

#pragma omp for nowait
  for (int i = 0; i < local_n; ++i) {
    if (visited[i] == 0 && dist[i] < t_min) {
      t_min = dist[i];
      t_v   = start_v + i;
    }
  }

  thread_mins[tid]     = t_min;
  thread_vertices[tid] = t_v;
}
```

`nowait` позволяет потокам не ждать конца итерации друг друга; безопасная запись в
`thread_mins[tid]` обеспечивается уникальностью индекса потока.

### 3.5 Ключевой фрагмент — OpenMP-релаксация рёбер

Файл: `all/src/ops_all.cpp`, функция `UpdateLocalDistances`.

```cpp
#pragma omp parallel for default(none) \
    shared(local_n, start_v, dist, visited, global_vertex, global_min)
for (int i = 0; i < local_n; ++i) {
  if (visited[i] == 0) {
    const InType global_i  = start_v + i;
    if (global_i != global_vertex) {
      const InType weight   = std::abs(global_vertex - global_i);
      const InType new_dist = global_min + weight;
      dist[i] = std::min(new_dist, dist[i]);
    }
  }
}
```

Каждый поток обрабатывает уникальный индекс `i`, поэтому гонок при записи в `dist[i]` нет.
Переменные `dist`, `visited`, `global_vertex`, `global_min`, `local_n`, `start_v` —
`shared`; `i`, `global_i`, `weight`, `new_dist` — приватные переменные цикла.

## 4. Детали реализации

- **Заголовочный файл:** `all/include/ops_all.hpp`.
- **Файл реализации:** `all/src/ops_all.cpp`.
- **Инициализация:** `PreProcessingImpl` устанавливает `dist_[0 - start_v_] = 0` на
  ранге-владельце вершины 0, всем остальным присваивает `INF`.
- **Число потоков OpenMP:** определяется динамически через
  `#pragma omp single { num_threads = omp_get_num_threads(); }` в начале `RunImpl`,
  что позволяет адаптироваться к конфигурации запуска.
- **Пользовательская операция MPI:** `MPI_Op_create` вызывается в каждом вызове
  `RunImpl` и освобождается через `MPI_Op_free` до выхода — утечек нет.
- **Синхронизация между уровнями:** `MPI_Allreduce` является коллективной операцией
  и неявно синхронизирует все ранги; барьеры OpenMP внутри каждого ранга обеспечивают
  согласованность записи в `thread_mins` до чтения `ReduceThreadMinima`.
- **Память:** $O(n / \text{size})$ для `dist_` и `visited_` на каждом ранге,
  $O(T)$ для буферов потоков, где $T$ — число потоков OpenMP. Матрица смежности
  не создаётся.
- **Директивы:** `default(none)` во всех параллельных регионах обеспечивает явную
  декларацию переменных.

## 6. Результаты

### 6.1 Корректность

ALL-версия прошла все функциональные тесты из `tests/functional/main.cpp` для
$n \in \{3, 5, 7\}$. Во всех случаях результат совпал с теоретическим значением
$\frac{n(n-1)}{2}$ и с результатами SEQ-версии. Тесты запускались при 2, 4 и 6
MPI-процессах — расхождений не наблюдалось.

### 6.2 Производительность

Базовая линия SEQ `task_run` составляет **0.01629 с**.
Ускорение вычислено как $S = T_{\text{seq}} / T_{\text{all}}$.

| Режим     | MPI-процессов | Время, с | Ускорение |
|-----------|---------------|----------|-----------|
| task\_run | 2             | 0.00448  | 3.64×     |
| task\_run | 4             | 0.08546  | 0.19×     |
| task\_run | 6             | 0.26532  | 0.06×     |
| pipeline  | 2             | 0.02010  | 0.81×     |
| pipeline  | 4             | 0.34253  | 0.05×     |
| pipeline  | 6             | 1.03576  | 0.02×     |

**Анализ.** При `task_run` с двумя MPI-процессами гибридная версия демонстрирует
ускорение 3.64× относительно SEQ. Это наилучший результат среди всех технологий
в данном сценарии, объяснимый сочетанием двух факторов: каждый из двух рангов
обрабатывает ровно половину вершин, а OpenMP-потоки внутри ранга дополнительно
ускоряют поиск минимума и релаксацию.

С ростом числа MPI-процессов до 4 и 6 производительность резко падает. Причина —
накладные расходы `MPI_Allreduce` с пользовательской операцией: для малых задач
($n = 200$) время межпроцессного обмена начинает многократно превышать полезную
вычислительную работу. Кроме того, каждый ранг запускает собственный пул потоков
OpenMP, и при 6 MPI-процессах на машине с 6 ядрами возникает перегрузка
(oversubscription): суммарное число потоков превышает число физических ядер,
что увеличивает накладные расходы на переключение контекстов. Режим `pipeline`
дополнительно усугубляет картину из-за оверхеда конвейерной инфраструктуры.

Таким образом, для данной задачи оптимальная конфигурация — два MPI-процесса
с несколькими потоками OpenMP в каждом.

## 7. Выводы

Гибридная ALL-реализация корректна и при конфигурации из двух MPI-рангов обеспечивает
наивысшее ускорение среди всех рассмотренных технологий — 3.64× в режиме `task_run`.
Объединение межпроцессного (MPI) и внутрипроцессного (OpenMP) параллелизма позволяет
эффективно задействовать аппаратные ресурсы при малом числе рангов. Масштабирование
ограничено накладными расходами `MPI_Allreduce` для задач малого размера: при $n = 200$
и числе рангов более двух коммуникационный оверхед превышает вычислительный выигрыш.
Для наблюдения полного потенциала гибридного подхода необходимо увеличить $n$ или
ограничиться одним рангом на узел с максимальным числом OpenMP-потоков.
