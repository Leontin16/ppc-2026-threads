## 1. Введение

Данный отчёт описывает реализацию алгоритма Дейкстры с применением технологии OpenMP.
Распараллеливание выполнено на двух ключевых этапах: поиске глобальной вершины с минимальным
расстоянием и релаксации рёбер. Последовательная версия (`SEQ`) используется как эталон
для проверки корректности и оценки ускорения.

## 2. Постановка задачи

Дан неориентированный полносвязный граф с $n$ вершинами.

- Вес ребра между вершинами $u$ и $v$ равен $|u - v|$.
- **Входные данные:** количество вершин $n$.
- **Выходные данные:** сумма длин кратчайших путей от вершины 0 до всех остальных.
- **Ограничения:** $n > 0$. Теоретическая сумма: $\frac{n(n-1)}{2}$.

## 3. Схема параллельного алгоритма

Алгоритм работает итерационно. Каждая итерация содержит три параллельных фазы:

1. **Параллельный поиск минимума** (`FindGlobalVertexOMP`): каждый поток независимо
   проходит свою часть вершин и находит локальный минимум. Затем в секции `#pragma omp single`
   поток-0 выбирает глобальный минимум и помечает найденную вершину как посещённую.
2. **Параллельная релаксация** (`RelaxEdgesOMP`): обновление расстояний для всех непосещённых
   вершин распараллеливается директивой `#pragma omp parallel for`. Запись в `dist[vertex]`
   безопасна, поскольку каждый поток обрабатывает уникальный индекс.
3. **Параллельное суммирование** (`CalculateTotalSumOMP`): финальная сумма расстояний
   вычисляется с редукцией `reduction(+ : total_sum)`.

### 3.1 Ключевой фрагмент — поиск глобальной вершины

Файл: `omp/src/ops_omp.cpp`, функция `FindGlobalVertexOMP`.

```cpp
// Каждый поток находит локальный минимум без явного lock
#pragma omp for nowait
for (int index = 0; index < n; ++index) {
    if (visited[index] == 0 && dist[index] < thread_min) {
        thread_min = dist[index];
        thread_vertex = index;
    }
}
local_min[thread_id]    = thread_min;
local_vertex[thread_id] = thread_vertex;

// Явный барьер синхронизует запись в local_min/local_vertex
#pragma omp barrier

// Один поток сводит локальные минимумы в глобальный и помечает вершину
#pragma omp single
{
    InType global_min = inf;
    for (int t = 0; t < num_threads; ++t) {
        if (local_min[t] < global_min) {
            global_min    = local_min[t];
            global_vertex = local_vertex[t];
        }
    }
    if (global_vertex != -1 && global_min != inf)
        visited[global_vertex] = 1;
    else
        global_vertex = -1;
}
```

`nowait` позволяет потокам не ждать конца цикла друг друга, а `#pragma omp barrier` затем
гарантирует, что все записи в `local_min` завершены до чтения в `single`.

### 3.2 Ключевой фрагмент — релаксация рёбер

Файл: `omp/src/ops_omp.cpp`, функция `RelaxEdgesOMP`.

```cpp
#pragma omp parallel for default(none) \
    shared(n, inf, global_vertex, dist, visited)
for (int vertex = 0; vertex < n; ++vertex) {
    if (visited[vertex] == 0 && vertex != global_vertex) {
        const InType weight   = std::abs(global_vertex - vertex);
        const InType new_dist = dist[global_vertex] + weight;
        dist[vertex] = std::min(dist[vertex], new_dist);
    }
}
```

Каждый поток записывает только в свой элемент `dist[vertex]`, поэтому гонок нет.
Классификация переменных: `dist`, `visited`, `global_vertex`, `n`, `inf` — `shared`;
`vertex`, `weight`, `new_dist` — приватные переменные цикла.

## 4. Детали реализации

- **Заголовочный файл:** `omp/include/ops_omp.hpp`.
- **Файл реализации:** `omp/src/ops_omp.cpp`.
- **Инициализация** выполняется в `PreProcessingImpl`: `dist_` и `visited_` выделяются в
  куче; `dist_[0] = 0`, остальные — `INF`. Размер задаётся в `ValidationImpl` ($n > 0$).
- **Число потоков** определяется динамически через `GetNumThreads()` (внутренняя обёртка
  над `omp_get_num_threads()`), что позволяет системе OpenMP использовать столько потоков,
  сколько доступно в данной конфигурации запуска.
- **Память:** $O(n)$ для `dist_`, $O(n)$ для `visited_`, $O(T)$ для буферов локальных
  минимумов, где $T$ — число потоков. Матрица смежности не создаётся.
- **Директивы:** `default(none)` во всех параллельных регионах обеспечивает явную
  декларацию переменных и исключает случайный `shared`.

## 6. Результаты

### 6.1 Корректность

Корректность OMP-версии проверена функциональными тестами из `tests/functional/main.cpp`
для размеров $n \in \{3, 5, 7\}$. Во всех случаях результат совпал с теоретическим значением
$\frac{n(n-1)}{2}$. Результаты идентичны SEQ-версии при числе потоков 1, 2 и более.

### 6.2 Производительность

Базовая линия SEQ `task_run` составляет **0.01629 с** (среднее по запускам с разным числом
MPI-процессов). Ускорение вычислено как $S = T_{\text{seq}} / T_{\text{omp}}$.

| Режим        | MPI-процессов | Время, с   | Ускорение | Эффективность |
|--------------|---------------|------------|-----------|---------------|
| task\_run    | 2             | 0.00633    | 2.57×     | —             |
| task\_run    | 4             | 0.01987    | 0.82×     | —             |
| task\_run    | 6             | 0.01602    | 1.02×     | —             |
| pipeline     | 2             | 0.02262    | 0.72×     | —             |
| pipeline     | 4             | 0.04794    | 0.34×     | —             |
| pipeline     | 6             | 0.07057    | 0.23×     | —             |

**Анализ.** При `task_run` с двумя MPI-процессами OMP демонстрирует ускорение 2.57×
относительно последовательной версии. С ростом числа MPI-процессов производительность
падает: 4 и 6 процессов делят одни и те же 6 физических ядер между несколькими MPI-рангами,
каждый из которых запускает собственный пул потоков OpenMP. Возникает перегрузка
(oversubscription), и накладные расходы на синхронизацию барьеров и переключение
контекстов начинают доминировать над полезной работой. Режим `pipeline` дополнительно
увеличивает накладные расходы конвейерной инфраструктуры, что объясняет его более низкие
показатели. Для задачи малого размера ($n = 200$) этот оверхед особенно заметен.

## 7. Выводы

OMP-реализация корректна и обеспечивает ускорение до 2.57× при оптимальном числе
MPI-процессов (2). Дальнейшее масштабирование ограничено числом физических ядер и
накладными расходами синхронизации на задачах малого объёма. Для получения полного
ускорения рекомендуется запускать ровно один MPI-процесс и задавать `OMP_NUM_THREADS`
в соответствии с числом доступных ядер.
